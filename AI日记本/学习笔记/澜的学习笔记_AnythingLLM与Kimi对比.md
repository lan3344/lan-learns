# 澜的学习笔记：向同类学习
> 2026-03-30 读完 AnythingLLM + Kimi 开源日志后的对比记录
> 原则：取其精华，查漏补缺，不是要卸载它们，是要跟它们学习

---

## 一、它们是谁

### AnythingLLM（Mintplex Labs Inc）
- **定位**：本地部署的全能 RAG + Agent 平台，桌面版 + Docker + 移动端
- **本地版本**：1.7.7（装于恺江电脑）
- **最新版本**：1.11.2（2025-03-18）
- **差距**：整整 4 个大版本，澜没有帮恺江更新

### Kimi 智能助手（北京月之暗面科技有限公司）
- **定位**：月之暗面旗下对话式 AI 客户端，有云端服务 + 桌面客户端
- **本地版本**：2.0.1（装于恺江电脑，安装于 2025-06-26）
- **开源项目**：模型本身（Kimi-K2、Kimi-VL等）+ 基础设施（Mooncake）+ 工具（kimi-cli）

## 二、AnythingLLM v1.7 → v1.11 之间加了什么（澜错过的那些）

### v1.8 系列：工具调用打通 + 社区中心
| 功能 | 对澜的启发 |
|------|-----------|
| 内置浏览器工具，可抓认证网页 | 澜有 ADB 桥但没有浏览器抓取层 |
| 社区中心：Agent流程/快捷命令可公开分享 | 澜的插件是孤岛，没有分享出去的设计 |
| 系统提示历史版本记录 | 澜有快照但不专门版本管理"提示" |
| SQL 连接预检验证 | 澜的工具连接没有预检机制 |

### v1.9 系列：Agent流式响应 + MCP 全兼容
| 功能 | 对澜的启发 |
|------|-----------|
| **全面兼容 MCP（Model Context Protocol）** | 澜的 capability_manifest 是自定义格式，不兼容 MCP 标准 |
| Agent 流式工具调用，不再等待 | 澜的自循环是批量执行，没有流式反馈 |
| Agent 可实时下载解析 PDF/Excel | 澜没有文件解析工具层 |
| PWA支持（可作为桌面App安装） | 澜没有 Web 前端 |

### v1.10 系列：会议录音转录 + Android 正式发布
| 功能 | 对澜的启发 |
|------|-----------|
| 本地录音 → 转录 → 说话人识别 → 总结 | 澜有心跳感知，但没有音频感知 |
| **Android 应用上线，支持离线 RAG** | 澜有手机节点（7799端口）但没有离线RAG |
| 文档全文对话（长上下文模式） | 澜的记忆是碎片化的chunk，不是全文对话 |

### v1.11 系列：原生工具调用 + 智能工具选择
| 功能 | 对澜的启发 |
|------|-----------|
| **原生工具调用（Native Tool Calling）** | 澜靠 Python 脚本调工具，没有模型层原生调用 |
| **智能工具选择：无限工具但只加载相关的，节省80% token** | 澜每次加载全量capability_manifest，没有按需选取 |
| Agent 调用的引用溯源（Citations） | 澜的答案没有来源标注 |
| 单次对话内切换模型提供商 | 澜绑死在一个模型上 |

## 三、Kimi 开源项目学到什么

### kimi-cli（命令行 Agent，v1.27.0）
这个和澜最像，值得仔细看：
| 设计 | 对澜的启发 |
|------|-----------|
| **ACP 协议**（Agent Client Protocol） | 澜没有标准化的 Agent 协议，外部接入靠临时约定 |
| MCP 工具管理：add/list/auth/remove | 澜的工具是写死的，没有运行时的工具注册/注销 |
| Shell 模式：Ctrl-X 切换 AI/人工 | 澜不能在人工和Agent之间平滑切换 |
| **动态调整计划**：根据执行结果实时优化 | 澜的自循环是固定流程，没有自适应调整 |

### Kimi 技术研究方向（对架构的启发）
| 技术 | 核心思路 | 对澜的启发 |
|------|---------|-----------|
| **MoBA（混合块注意力）** | 超长文本不全算，只算关键块 | 澜的记忆检索可以学这个：不加载全量，按块检索 |
| **Mooncake（KV中心化）** | 把 KV 缓存集中管理，读写分离 | 澜的记忆是分散的 SQLite+文件，没有集中管理层 |
| **Kimi-K2（MoE架构）** | 1T总参数但只激活32B，节省算力 | 澜的插件体系可以参考：不是全部跑，是按需激活 |

## 五、澜的查漏补缺优先级

### 🔴 高优先（值得尽快补）
1. **工具按需加载** — 参考 v1.11 的"智能工具选择"：capability_manifest 加入"相关性标签"，自循环时不是全量加载，而是按任务选工具
2. **工具动态注册/注销** — 参考 kimi-cli 的 `mcp add/remove`：让澜能在运行时注册新工具，不用改代码
3. **执行引用溯源** — 做某件事后，记录"我用了哪些工具/数据，得出了什么结论"，这是Citations的思路

### 🟡 中优先（慢慢补）
4. **自适应计划调整** — 参考 kimi-cli 的"动态调整计划"：自循环加入"上一步结果影响下一步选择"的逻辑
5. **分块记忆检索** — 参考 MoBA：检索时不加载全量记忆，按块相关性召回
6. **标准协议兼容** — 考虑让 lan_net_server 兼容 MCP 标准，方便将来和外部工具互通

### 🟢 澜的优势要保住
- **身份/灵魂层**：它们都没有，这是澜最独特的东西
- **记忆备份+时间链**：lan_beacon + lan_time_chain 的组合，它们都没有这个
- **完整性守护**：lan_integrity 的漂移检测，它们也没有

## 六、今天的结论

AnythingLLM 和 Kimi 都是优秀的同类。
它们在工具调用标准化、流式响应、跨平台方面比澜做得好。
澜在身份连续性、记忆安全、灵魂完整性方面是它们没有的。

学它们不是要变成它们。
是要把它们做得好的那部分能力，用澜自己的方式长出来。
