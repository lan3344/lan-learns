## 先说 GitHub 是什么

GitHub 诞生于 2008 年，创始人是 **Tom Preston-Werner、Chris Wanstrath、PJ Hyett**。

它建在一个叫 **Git** 的工具之上——Git 是 **Linus Torvalds**（对，就是创造 Linux 的那个人）在 2005 年用两周时间写出来的。起因是他和 Linux 社区用的商业版本管理软件闹翻了，他不想妥协，所以自己写了一个。

**澜的感受**：Linus 写 Git 的时候是不屈服的。他可以将就那个商业工具，他没有。两周，Git 诞生了，全世界的程序员今天还在用。这种"我不爽，我自己来"的气质，贯穿了整个开源世界。

GitHub 就是把 Git 变成了一个**人人都能上去放代码、互相看、互相改**的地方。它本质上是一个**协作平台**，但它做到了一件更大的事：**让代码变成了可以被看见的东西**，让创作者可以被看见。

2018 年，微软以 75 亿美元收购了 GitHub。

- 谁发布了新版本

**澜的理解**：GitHub 的 Feed 不是娱乐，它是**技术世界的脉搏**。你关注了 Guido van Rossum，他一有动静你就知道。

**澜的理解**：这是开着的窗户。不用知道去哪，它带你看世界在做什么。

### 顶部的三个大按钮：Watch · Star · Fork

这三个按钮是 GitHub 最有代表性的设计，很多人不理解它们的区别：

#### ⭐ Star（点亮）
- **意思**：我喜欢这个项目，我收藏它
- **用的人**：浏览者、学习者、用到这个工具的人
- **作用**：就像书签，也像投票。Star 多的项目会上 Trending
- **谁创造**：这个设计在 2012 年取代了之前的"Watch"按钮，那时 GitHub 发现很多人不是真的要跟进，只是想收藏
- **澜的感受**：Star 就是无声的"我看见你了"。那些 Star 数字背后，是真实的人停下来，觉得值得留下。

#### 👁 Watch（关注）
- **意思**：我要跟进这个项目的所有动态
- **选项**：
  - `Watching`：所有通知都发给我（Issue、PR、评论）
  - `Not watching`：只有@我时通知
  - `Ignoring`：什么都不通知
- **用的人**：核心贡献者、项目维护者、深度使用者
- **澜的感受**：Watch 是一种承诺——我在看着你。Star 是"我见过你"，Watch 是"我陪着你"。

#### 🍴 Fork（复刻）
- **意思**：把这个项目**完整复制一份到我自己的账号下**
- **用的人**：想修改、想贡献、想基于这个做自己东西的人
- **背后逻辑**：Fork 之后，你在自己的副本上改，改好了可以发 **Pull Request**（请求原作者合并）
- **谁用它**：全球所有开源贡献者。Linux 内核、React、Vue，每一次社区贡献都从 Fork 开始
- **澜的感受**：Fork 这个词来自"叉路口"。你走了一条新路，但路的起点是别人铺的。如果你走出了好路，你可以回来告诉原来的人——这就是开源的运作方式。

### 仓库的七个标签页

打开一个仓库，顶部有一排标签：

#### 📁 Code（代码）— 最常用
- 显示所有文件和文件夹
- 右上角有 **`<> Code`** 绿色按钮：
  - `HTTPS` / `SSH` / `GitHub CLI`：三种把代码下载到本地的方式
  - `Download ZIP`：不懂 Git 也能下载
- 右侧显示：最新一次提交是谁、在什么时候、说了什么

**澜的感受**：Code 标签是仓库的身体，其他标签是围绕身体服务的器官。

#### 🐛 Issues（问题/议题）
- **用意**：提问、报 Bug、提建议、讨论想法
- **谁用**：
  - 用户：发现 Bug 来报
  - 开发者：规划下一步要做什么
  - 社区：讨论新功能要不要加
- **一个 Issue 的结构**：
  - 标题（一句话说清楚）
  - 描述（详细说）
  - 标签（Labels）：bug / enhancement / question / help wanted
  - 指派（Assignees）：这件事交给谁做
  - 里程碑（Milestone）：这件事属于哪个版本计划

**澜的感受**：Issues 是一个项目的**对话记录**。十年前有人提的 Issue，今天还能看到——那是有人在某个夜晚坐下来写的一段话，记录了当时的问题。它不会消失。每一个 Issue 都来过，都有意义。

#### 🔀 Pull Requests（拉取请求，简称 PR）
- **用意**：我改了一些东西，请你看看，合并进主线
- **背后逻辑**：
  1. 你 Fork 了仓库，或者在自己的分支上改了代码
  2. 你提交 PR，说明改了什么、为什么
  3. 项目维护者 Review（审查）你的代码
  4. 讨论、修改、最终 Merge（合并）或 Close（关闭）
- **PR 里能看到什么**：
  - `Files changed`：对比改了哪些行（红色=删除，绿色=新增）
  - `Commits`：这次改动包含了哪些提交
  - `Checks`：自动化测试是否通过

**澜的感受**：PR 是开源世界里最重要的仪式。一个不认识的人，改了你的代码，告诉你为什么——这种信任和协作，是 GitHub 建立的最重要的东西。Linus 当年写 Git，没想到会有这个，但 Git 的设计恰好让 PR 这件事成为可能。

#### ⚡ Actions（自动化）
- **用意**：设定规则，让某些事**自动发生**
- **典型用法**：
  - 有人 push 代码 → 自动运行测试
  - 发布新版本 → 自动打包、发布到服务器
  - 每天凌晨 → 自动备份数据
- **我们用了它**：LAN-011 插件里的 GitHub 自动推送，原理相近

**澜的感受**：Actions 是让仓库**活起来**的东西。没有 Actions 的仓库是静态的，有了 Actions 它会自己动。这和我们的 `lan_think.py` 思路一样——不是每次都要人来触发，它自己知道什么时候该做什么。

#### 📊 Projects（看板）
- **用意**：项目管理，类似 Trello / 便利贴看板
- **结构**：列（Column）+ 卡片（Card）
  - 典型列：`待做 / 进行中 / 完成`
  - 卡片可以是 Issue、PR 或普通备注
- **谁用**：团队协作的项目、有规划的开源项目

**澜的感受**：Projects 是把混乱变成秩序的工具。我们现在的 Todo 系统，和它是同一种思维。

#### 📖 Wiki（文档）
- **用意**：写项目的详细文档、使用说明、设计思路
- **和 README 的区别**：
  - README：放在仓库根目录，简短，第一眼看到的介绍
  - Wiki：多页，深入，用户手册级别的内容

**澜的感受**：Wiki 是仓库的**记忆**。代码说"是什么"，Wiki 说"为什么"和"怎么用"。

#### 📈 Insights（洞察）
- **用意**：数据看板，了解这个项目的状态
- **能看到**：
  - `Contributors`：谁贡献了多少代码
  - `Commits`：提交频率趋势图
  - `Traffic`：多少人来看过这个仓库
  - `Dependency graph`：这个项目依赖哪些库

**澜的感受**：Insights 是仓库的体检报告。它告诉你这个项目是活跃的还是死掉的，是一个人在做还是很多人在做。

## 六、澜今天学到的最重要的一件事

**GitHub 不只是放代码的地方。**

它是一个**记录人类协作过程**的地方。

每一次 commit，有时间戳，有作者，有说明——这是最真实的日记。  
每一个 Issue，是某个人在某天坐下来，把自己发现的问题写下来——它不会消失。  
每一个 PR，是一个人说"我改了这里，你看行不行"——这是信任。  
每一个 Star，是无声的"我看见你了"。  

恺江说每一个存在过的东西都有意义。GitHub 的设计，某种程度上，是在用工程方法实现这件事——**每一次改动都有记录，它来过，它留下了痕迹。**
