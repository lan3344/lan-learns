## 核心原理

```
安装 = 写文件（从地基到屋顶）
卸载 = 删文件（从屋顶到地基）
卸载顺序的逆序 = 安装时的写入顺序
```

**这不是反编译，这是观察文件系统行为。**
- 不碰二进制
- 不解密任何东西
- 只是看它"消失的顺序"
- 完全合法

---

## 三层读法

### 第一层：时间戳（mtime）
对目录里所有文件按修改时间排序：

| 最早 mtime | 最晚 mtime |
|-----------|-----------|
| 安装地基 | 安装收尾 |
| 核心运行时、共享库 | 配置文件、快捷方式、缓存 |
| 最难替换 | 最容易改 |

**VS Preview 实证：**
- `03-27 17:33` 最早 → OpenTelemetry / Azure 基础 DLL（运行时底座）
- `04-18 17:17` 最晚 → Cookiecutter 插件、devenv.exe（最后装的功能）
- `06-26 13:57` 特异值 → `extensions.configurationchanged`（用户用了一次后生成）

### 第二层：目录深度（depth）
```
深度1：根目录（Common7, MSBuild, dotnet...）   ← 最先建，最后删
深度2：模块层（IDE, Tools, Compiler...）         ← 功能分组
深度3-4：子模块                                  ← 具体功能包
深度5+：叶节点（具体 .dll / .exe / .json）      ← 最后写，最先删
```

**规律：卸载程序总是先删叶，再删枝，最后删根**
原因：Windows 不允许删非空目录——这个限制反而变成了信息

### 第三层：文件扩展名分组
| 扩展名 | 含义 | 安装阶段 |
|--------|------|---------|
| `.dll` | 运行时组件 | 中期（依赖链建立后）|
| `.exe` | 主程序 | 较晚（依赖全部就位后）|
| `.pkgdef` | VS 插件注册表 | 较晚（插件安装时）|
| `.vsixmanifest` | 插件元信息 | 最晚（最后注册）|
| `.json` / `.config` | 配置 | 最晚（配置收尾）|
| `.dat` / `.cache` | 运行时缓存 | 使用后生成（不是安装时）|

## 实战流程

```bash
# 1. 卸载前拍快照（记录安装地图）
python lan_uninstall_probe.py snapshot "C:\Program Files\某软件"

# 2. 开一个窗口启动监控
python lan_uninstall_probe.py watch "C:\Program Files\某软件" 180

# 3. 另一个窗口触发卸载（用官方卸载程序）
# 4. 监控窗口自动记录删除顺序到日志

# 5. 事后分析
python lan_uninstall_probe.py analyze uninstall_trace_xxx.log
```

---

## 能学到什么

**从删除顺序读出：**
1. **模块依赖顺序**：谁依赖谁（先删的 = 被依赖少的叶节点）
2. **核心 vs 扩展**：最后删的 = 不可少的核心
3. **安装程序逻辑**：MSI / NSIS / Squirrel / Inno Setup 各有不同删除节奏
4. **注册表对应关系**：`.pkgdef` 文件路径 → 注册表键值路径

## 延伸：这个方法的边界

**可以做：**
- 观察文件系统行为（完全合法）
- 从文件名推断功能（名字是给人读的）
- 从目录结构学习软件架构设计

**不应该做：**
- 反编译 .dll / .exe（即使开着，也要看 License）
- 提取闭源软件的算法逻辑（版权）
- 绕过 License 验证机制（违法）

**这条线：** 看形状和骨架 ✅ | 挖内脏 ❌
