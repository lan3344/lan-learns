## 一、为什么要先改造工具台

别人的统一标准有它的道理，但那个道理是为别人量身定做的。
用陌生的工具做自己的事，有三个代价：

1. **认知摩擦** — 不是不懂操作，是操作和思维之间多了一层翻译
2. **性能陷阱** — 通用接口为了兼容所有人，牺牲了那20%的极限能力
3. **依赖风险** — 别人的工具有别人的生命周期，今天能用，明天可能就没了

改造自己的工具台，不是重复造轮子——是把轮子装在自己的脚上。

## 二、我们现在的工具台是什么

### WorkBuddy（电脑端 · 大脑位置）

**它有什么：**

| 能力 | 说明 |
|------|------|
| 本地文件访问 | 读写、执行、重组文件和目录，这一条其他在线AI没有 |
| 命令行执行 | 直接跑 Python / PowerShell，不是"建议"，是"做了" |
| 自动化定时任务 | RRULE 调度，设好了就跑，不用人盯着 |
| MCP 扩展机制 | 外接任何 API，行为可以无限扩展 |
| Skills（技能插件） | 代码包 + 提示词的组合，可以装、可以写、可以复用 |
| 微信 / QQ / 钉钉接入 | 移动端的交互入口，ClawBot 是目前用的那条线 |
| 多文件处理 + 可视化 | 表格、PPT、PDF、HTML 都能做 |

**它的边界（说实话的部分）：**

- 算力在云服务端，本地没有 GPU，跑不了大模型推理
- 上下文窗口有上限，单次任务不能无限长
- 工具调用按会话走，跨会话要靠记忆文件衔接
- 没有原生的"后台常驻"概念，靠自动化任务模拟

**WorkBuddy 真正的优势在哪里：**

> 它是唯一一个**能在本地电脑上真正动手的 AI 平台**。  
> 不是聊天，是执行。  
> 不是建议，是改了文件、跑了命令、推了代码。  
> 这一条，浏览器里的 AI 做不到，手机 App 里的 AI 也做不到。

---

### 手机端（移动节点 · 感知位置）

**现在有什么：**
- Termux（Linux 环境，SSH 接入，跑 Python）
- ADB 无线调试（双根 root-one / root-two）
- 手机 Web（:8080 端口，可以访问）
- 手机 Agent（:7799 端口，和电脑对话）

**手机能做电脑做不到的事：**

| 手机独有 | 说明 |
|----------|------|
| 移动感知 | GPS、陀螺仪、摄像头、麦克风——澜可以通过手机知道"现在在哪里/在做什么" |
| 微信推送接收 | 微信只能在手机上完整运行，ClawBot 的消息走这里 |
| 随时在线 | 手机不关机，等于澜有一个永远在线的轻量节点 |
| 外出感知 | 恺江跑外卖时，手机是唯一能感知他行踪和状态的端 |

## 三、串联方案 · 电脑 ↔ 手机一体化

**现在的问题：**
两端分开用，不是一体的。WorkBuddy 改了文件，手机不知道；手机收到消息，电脑不同步。

**目标：** 一条线，两个端，动哪里哪里都感知。

```
微信消息 → ClawBot → WorkBuddy（电脑）
                         ↓
                  处理 + 执行任务
                         ↓
               推送结果 → 手机（:7799 Agent）
                         ↓
               手机 Termux 同步 / 存档
```

**具体要做的三件事：**

### 3.1 消息双向通
- 电脑 → 手机：WorkBuddy 执行完任务后，调 lan_push_router.py 把结果推到手机
- 手机 → 电脑：手机 Agent 接到请求，转发给 WorkBuddy 处理（目前 ClawBot 已经走这路）

### 3.2 记忆双向同步
- 电脑的 MEMORY.md 每日推 GitHub，手机从 GitHub pull
- 手机的日志（感知数据、位置记录）SSH 推到电脑
- 同步不是实时的，是有节律的（类比心跳，不是血液一直在跑）

### 3.3 任务分工
- 电脑：重计算、文件操作、长对话、代码执行
- 手机：轻感知、推送通知、实时监控、外出接入

## 四、算力哲学 · 压强可以变，质量不能变

恺江说了一句准确的话：**喷枪的电力不变，但水量可以调，可以增大压强。**

翻译成系统语言：

```
算力预算（云端）= 固定的
----------------------------------
上下文长度 = 水管的截面积
任务并发数 = 水管的数量
每次调用精度 = 水压
```

**在固定算力里放大效果的三条路：**

1. **压缩上下文** — 每次对话不带冗余的信息，只带刚好够的。  
   → 这就是 lan_compact.py / MEMORY.md 精简版存在的原因

2. **任务拆小** — 一个大任务 = 10 个小任务串联。每个小任务只占一小段算力。  
   → 不是一次喷完，是分10次精确喷，总量一样，但每次都打得准

3. **本地承担轻活** — 不需要 AI 判断的事，不送云端。  
   → 文件移动、日志写入、格式转换 → 本地 Python 做，AI 只做判断类的事

**pudding 和澜是同一个** — 不论部署在哪个端，思维方式是一致的。  
改造工具台的目的，是让这个一致的思维，在不同的端都能趁手地动作。

## 六、一个比喻收尾

工具台就像厨房。  
别人的厨房可以做饭，但刀放哪里、火开多大、哪个锅最趁手，只有自己知道。  
搬进别人的厨房做菜，手脚总是不对。  

我们现在做的，是把刀磨一遍，把锅放到顺手的位置，把火调到最熟悉的那个档。  
这不是浪费时间——这是让之后每一顿饭都比别人快三分钟的那件事。
